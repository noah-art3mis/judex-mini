# HC backfill — year-priority gap-sweep plan

**Status: revised 2026-04-18.** Replaces both the prior per-year single-worker "Tracks 1/2/3" structure and my earlier "full-year + `--resume`" draft from this session. See `git log` for history.

## Decisions (2026-04-18)

- **Gap-sweep, not full-year.** Each year's CSV is generated by filtering the year's HC ID range against what's already on disk under `data/cases/HC/`. Only uncaptured IDs are fetched. Saves ~30% of wall vs. full-year + `--resume` and sidesteps the state-aggregation step.
- **Skip paper era.** Plan runs tiers 0–13 (years 2026 → 2013). Pre-2013 coverage stays as-is (shard-3 in the prior run captured 1971–1990 at ~400–800/year; 1991–2012 is left as a known gap, not this sweep's job).
- **Catch-up 2026, don't skip it.** Tier 0 is a small catch-up sweep (~917 IDs) so the current year stays current.
- **Stopped the prior in-flight run on 2026-04-18 20:44 UTC.** Archived at `runs/archive/2026-04-17-hc-full-backfill-sharded/`; REPORT at [`docs/reports/2026-04-17-hc-full-backfill-sharded.md`](reports/2026-04-17-hc-full-backfill-sharded.md).
- **8 shards per year, not 4.** A 44-proxy batch landed on 2026-04-18 and was split + rebalanced across `proxies.{a..h}.txt` at **10 proxies per pool** = **80 sessions per year-sweep**. 4 extras held in `config/proxies.reserve.txt` for future use. Wall estimates **roughly halve** but are empirically untested — the portal may have a rate ceiling below 2× at this concurrency. Tier 0 validates.

## Why this shape

The archived `2026-04-17-hc-full-backfill-sharded` sweep had 4 shards over the full HC ID space. It worked, but produced uneven per-year capture (shard-1 at HC 138 722 solo-bottlenecked, shard-2 at HC 121 057 with 53 k of untouched 2001–2012 territory, shards 0 and 3 near their range floors at stop time).

Switching to year-priority 8-shard, newest-first, makes the per-year capture curve crisp and research-priority-ordered. Eight proxy pools (`config/proxies.{a,b,c,d,e,f,g,h}.txt`) run in parallel, but their union of work is one year at a time, not 8 disjoint ID slabs across the full corpus.

## Per-year mechanics

For each tier `YYYY`:

1. **Derive year ID range** from `hc_calendar.year_to_id_range(YYYY)`.
2. **Generate gap CSV** — enumerate `[lo, hi]`, emit only rows whose `processo_id` is NOT present under `data/cases/HC/judex-mini_HC_<N>-<N>.json`.
3. **Shard 4-way** — `scripts/shard_csv.py --shards 4`.
4. **Launch 4 workers** via `scripts/launch_hc_year_sharded.sh $YYYY` — disjoint proxy pools `a/b/c/d`, disjoint shard CSVs, detached via `nohup … & disown` (see `CLAUDE.md § Surviving session death`).
5. **Probe + monitor** — `scripts/probe_sharded.py`; cron monitor via `/schedule` for dead-worker / collapse / stall alerts.
6. **Wait for completion** — all 4 shards exit when their CSVs are exhausted.

Seeding `sweep.state.json` from a prior run is **not needed** — the gap CSV contains only uncaptured IDs by construction. This is the big simplification over the full-year-plus-resume approach.

## Stop criteria (per year)

Halt and diagnose before launching the next tier if any of:

- `ok / total < 0.80` for the year's gap CSV. (Lower bound assumes ~80–90% density of real HCs in the range; paper-era concerns don't apply since we're not sweeping below 2013.)
- **Calendar drift gate:** < 80% of captured cases have `data_protocolo` year within `YYYY ± 1`.
- Circuit-breaker abort in any shard (`sweep.state.json` status = `aborted`).
- New error class (403 / 429 / 5xx) in `sweep.errors.jsonl`.
- Wall > 2× the table estimate.

## Post-year routine

1. **Read** `runs/active/<date>-hc-$YYYY/shard-*/sweep.state.json`.
2. **Consolidated REPORT** — promote to `docs/reports/<date>-hc-$YYYY.md`.
3. **Refresh `judex/utils/hc_id_to_date.json`** — append new `(processo_id, data_protocolo)` anchors from the captured cases. Tightens the next tier's range derivation.
4. **Archive** `runs/active/<date>-hc-$YYYY/` → `runs/archive/`.
5. **Launch next tier** if gates passed and ScrapeGW quota ≥ 1 GB.

## Interaction with the in-flight 4-shard backfill

**Decision: let it keep running until the year-launcher is ready to fire.** Stopping it now idles 4 proxy pools for the hour or so it takes to write the launcher + gap-CSV generator. The in-flight shards are still adding to `data/cases/HC/`, which shrinks the gap numbers we quote above.

**Stop procedure (only when tier-0 is ready to launch):**
```bash
xargs -a runs/active/2026-04-17-hc-full-backfill-sharded/shards.pids kill -TERM
# wait for clean shutdown (circuit-breaker + final report writes)
pgrep -af "run_sweep.*hc_full_backfill_shard"   # verify no children linger
mv runs/active/2026-04-17-hc-full-backfill-sharded runs/archive/
```

All atomic-state contracts in `judex/sweeps/process_store.py` guarantee zero data loss on SIGTERM. The captured HCs stay on disk under `data/cases/HC/`.

## Launch scripts (to be written)

### `scripts/generate_hc_year_gap_csv.py`

Produces the gap CSV for one year. Essentially:

```python
# args: --year YYYY --out tests/sweep/hc_YYYY_gap.csv
import csv, argparse
from pathlib import Path
from judex.utils.hc_calendar import year_to_id_range

def have_ids(hc_dir: Path) -> set[int]:
    out: set[int] = set()
    for f in hc_dir.glob('judex-mini_HC_*.json'):
        stem = f.stem.replace('judex-mini_HC_', '')
        if '-' in stem:
            lo_s, hi_s = stem.split('-', 1)
            try:
                for n in range(int(lo_s), int(hi_s) + 1):
                    out.add(n)
            except ValueError:
                pass
    return out

# year_to_id_range(year) → (lo, hi)
# emit rows (classe, processo) in reverse (newest first) for IDs not in have_ids
```

Walks `data/cases/HC/`, derives year range, emits uncaptured IDs in descending order. Running it with `--year 2025` produces `tests/sweep/hc_2025_gap.csv` with ~6 857 rows.

### `scripts/launch_hc_year_sharded.sh`

Parameterized fork of `scripts/launch_hc_backfill_sharded.sh`:

```bash
#!/usr/bin/env bash
# Usage: nohup ./scripts/launch_hc_year_sharded.sh 2025 > launcher-hc-2025.log 2>&1 & disown
set -uo pipefail
cd "$(dirname "$0")/.."

YEAR=${1:?Usage: $0 YYYY}
DATE=$(date +%Y-%m-%d)
ROOT="runs/active/${DATE}-hc-${YEAR}"
CSV="tests/sweep/hc_${YEAR}_gap.csv"
SHARD_DIR="tests/sweep/shards/hc_${YEAR}"
PROXY_FILES=(config/proxies.a.txt config/proxies.b.txt config/proxies.c.txt config/proxies.d.txt)

mkdir -p "$ROOT" "$SHARD_DIR"

# 1. Generate gap CSV if missing
[[ -f "$CSV" ]] || PYTHONPATH=. uv run python scripts/generate_hc_year_gap_csv.py \
    --year "$YEAR" --out "$CSV"

# 2. Shard 4-way
PYTHONPATH=. uv run python scripts/shard_csv.py --csv "$CSV" --shards 4 --out-dir "$SHARD_DIR"

# 3. Launch 4 workers
: > "$ROOT/shards.pids"
for i in 0 1 2 3; do
    SHARD_OUT="$ROOT/shard-$i"; mkdir -p "$SHARD_OUT"
    nohup env PYTHONPATH=. uv run python scripts/run_sweep.py \
        --csv "$SHARD_DIR/hc_${YEAR}_gap.shard.$i.csv" \
        --label "hc_${YEAR}_shard_$i" \
        --out "$SHARD_OUT" \
        --items-dir data/cases/HC \
        --proxy-pool "$PWD/${PROXY_FILES[$i]}" \
        --resume \
        >> "$SHARD_OUT/driver.log" 2>&1 &
    echo "$!" >> "$ROOT/shards.pids"
done
```

No state seeding — the gap CSV already excludes captured IDs.

## What this plan does NOT cover

- **Paper era (pre-2013).** Explicitly out of scope. The 1991–2012 capture gap identified in the 23:20 UTC observation stays unfilled by this sweep. Revisit as a separate class/era plan if research value emerges.
- **Non-HC classes** (RHC, AP, MS, RE, ADI). One class at a time. Reprioritize after HC is done.
- **PDF download** — `scripts/baixar_pecas.py` (writes raw bytes to `data/cache/pdf/<sha1>.pdf.gz`); run on the completed HC corpus.
- **Text extraction / OCR** — `scripts/extrair_pecas.py --provedor {pypdf|mistral|chandra|unstructured}` over the downloaded cache; no STF fetch.
- **Robots.txt posture** — unchanged.

## Rollback

If per-year sweeps show worse WAF behavior than the full-range 4-shard (unlikely — same total ID diversity, same proxy rotation), revert to `scripts/launch_hc_backfill_sharded.sh` over `hc_all_desc.csv`. The year-sharded launcher is additive and doesn't touch the existing launcher.
