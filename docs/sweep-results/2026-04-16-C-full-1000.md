hypothetical; the 403 wall means we cannot actually complete a
1000-ADI sweep from one IP today.

## STF's current block threshold

- **~107 consecutive processes** at this scraper's concurrency
  (single driver, 4 tabs in parallel per process, plus PDF fetches
  from a different origin). Sweep B's 50 fit comfortably; 108 did
  not.
- **Response signature**: `HTTP 403 Forbidden` on `/processos/*`
  (both `listarProcessos.asp` and tab fetches), **not** 429. The
  existing tenacity retry policy does not retry 4xx other than 429,
  so the scraper correctly stops hammering — but it also has no
  way to know when the ban lifts. Empirically, STF's portal bans
  appear to be minutes-long rolling windows.

## Error-type breakdown

The stable driver logs errors as a single `error` string
`"{ExceptionClass}: {message}"`. Post-processing the log:

| error type | HTTP status | count | endpoint                       |
|------------|------------:|------:|--------------------------------|
| HTTPError  | 403         | 892   | `listarProcessos.asp`          |
| HTTPError  | 403         | 1     | `abaRecursos.asp` (mid-process) |

Every failure in this sweep is the same error. No TimeoutError,
ConnectionError, or 5xx observed.

**Gap the user flagged on this sweep:** the error is stored as one
string field. A structured record would separate `error_type`
(class name), `http_status` (int), `endpoint`, and `error_msg`.
Ticket: enrich `AttemptRecord` in `scripts/sweep_state.py` with
those fields when finishing the robust integration.

## What this means for next steps

- **Robots.txt posture is no longer theoretical.** The handoff
  flagged `robots.txt` as unresolved; this sweep showed STF enforces
  it with a 403-based block at ~107 processes. Options from the
  handoff still stand:
  1. `User-Agent` + `NOTICE.md` + `Ouvidoria` contact
  2. Talk to STF first for whitelisted access
  3. Ship as a parser, users populate `.cache/` themselves
- **Handoff step #3 ("429-aware backoff") needs broadening** to
  "403-aware backoff" too. 429 is polite-throttle; 403 is ban. The
  tool should detect the 403 wall, stop, and surface a clear
  message rather than fast-fail through hundreds of rows.
- **The append-log + retry-from system** (in-progress at
  `scripts/sweep_state.py`, currently stashed) is exactly what's
  needed to pick up from a sweep like this: 893 entries in
  `sweep.errors.jsonl`, a cooldown period, then
  `run_sweep.py --retry-from ...` resumes just those.
- **No new schema surprises** in the 107 that succeeded. HC
  coverage from sweep A plus ADI variety from this sweep's first
  107 gives reasonable confidence that the HTTP parsers hold up
  across the ADI class.
